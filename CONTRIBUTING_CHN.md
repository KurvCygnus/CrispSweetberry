# 贡献指南

## 在开始之前

欢迎来到澄莓物语的代码风格和文档化指南! 本指南将帮助您了解我们在项目中遵循的编码标准. 如果您有任何问题, 请和我们讨论.

*这些指导原则优先考虑长期维护性, 而不是贡献速度. 如果你正在寻找一个快节奏或实验性的修改环境, 这个项目可能不适合你*.

## 工具与环境

1. 我们强烈建议使用 **IntelliJ IDEA** 进行开发.
   由于 NeoForge 在 VSCode 下的工具链支持仍不完善，目前仅对 IntelliJ IDEA 提供官方开发指导.

2. 如果你在使用**IntelliJ IDEA**开发, 请安装 <u>[**Better Highlights**](https://plugins.jetbrains.com/plugin/12895-better-highlights)</u> 插件, 用于高亮项目中的重要注释.
   我们推荐使用 **`*` 作为单行注释前缀来标注必须注意的关键信息**和**板块的功能主题**(对于板块, 可以在[这里](#region)了解),
   **`!` 来标识代码中的潜在错误、边界情况或复杂的异常处理流程.**

   *示例:*

   ```java
   //e.g.
   //* 这是一个重要的注释!
   //! 这是一个解释潜在错误的注释.
   ```

3. 所有与空值相关的注解必须使用 **JetBrains** 的注解库(如 `@NotNull` / `@Nullable`), 以避免不同注解体系混用导致的歧义.

## 代码风格与文档要求

1. **我们以"零警告(zero warnings)"为目标**.
    对于无法避免的 API 设计问题例如 `datafixer` 参数，一般传入 `null`, 但我们建议这样配合注解和解释性注释:

   ```java
   @SuppressWarnings("ConstantConditions")//* 原因及其解释...
   SomeObject.someFunc(null);
   ```

2. 我们不介意你个人的代码习惯, 但是请确保你的**命名风格是驼峰命名法**, 其它可以通过格式化解决的我们完全接受.

3. 你修改或新增的功能, 必须配套编写 Javadocs, 文档应当说明:
   - 这个功能*为什么存在*
   - 如果不能从功能实现上轻易理解, 它该*如何被使用*
   - 如果功能带有安全隐患设计(*footgun*)或者设计妥协(*trade-off*), 有哪些*应当避免的用法*

   > 我们更偏好简洁、准确、可读性强的文档, 而非冗长描述.

   `@param`/`@throw`/`@return`并非强制, 但我们强烈建议保留:
   - `@since + [模组版本]`
   - `@author + [你的名字]`

   这是你的贡献, 请为它署名.

4. 请优先考虑可读性, 而非代码长度.
   即使类名、方法名或变量名较长, 只要含义清晰, 就是可接受的.
   对于常量, 你应该把它们放在使用的类中, 只有在常量通用的情况下放在`CrispConstants.java`中, 并放在对应的内部类中.

5. 我们在乎代码的结构. 我们建议将功能相似的字段/方法分组到一起, 并使用以下的方法来组织代码:

   <a id="region"></a>

    ```java
    //  region
    //* 板块类型

    //字段, 构造方法, 方法...

    //endregion
    ```

    **这对任何阅读源代码的人都会有很大的帮助**.

6. 对于简单/不复杂的代码, 尤其是一些简单的继承实现, 我们不做出以上声明的强制要求, 但必须遵守第4条规则, 即保持代码的整洁和一致性.
   *但也不要忘了`@since + [模组版本]`和`@author + [你的名字]`, 这是你的贡献*.

7. 如在这些方面有疑问, 最终标准以核心维护者的判断为准.

## 协作与沟通

1. 如需重构他人的代码, 请先进行讨论.
   **在未沟通的情况下直接修改他人实现, 容易造成不必要的冲突, 也会降低代码历史的可追溯性**.

2. 开发模组不应只是关于模组本身.
   讨论不必局限于纯粹的实现细节, 我们重视人与人之间的交流——模组开发者不是机器人.

3. 请在讨论时注意你的言行. 不当的言行对你自己和他人都是不好的.
   屡次不尊重他人的行为可能会导致讨论终止或贡献被拒绝, 最终决定权归核心维护者所有.

## 关于提交

1. 设计哲学解释和最终的功能决策由核心维护者(们)保留, 即使建议是合理或实施得当的.

2. 如果一个功能引入了新的必知, 进阶步骤或者配置项给普通玩家, 我们不会认同这是一个恰当的QoL.

> 如果你对某段设计有疑问、不同看法, 或者认为有更好的实现方式, 欢迎提出讨论. 澄莓物语希望成为一个能让人学习、理解、并长期维护的项目.
